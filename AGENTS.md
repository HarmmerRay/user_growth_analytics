# AGENTS — Coding Agent 规范

本文件约束在本仓库内工作的 Coding Agent。默认遵守；与用户当次指令冲突时，以用户指令为准。

## 总则

1. **先读后写**：改代码前先阅读相关文件与 `PROGRESS.md`，确认当前进度与约定。  
2. **最小改动**：只改完成任务所需的代码，不做无关重构、不顺手「优化」周边。  
3. **与现有风格一致**：命名、目录、Frappe 惯例跟随仓库已有实现。

## 复用优先

- **封装新函数 / 类 / 模块之前，先搜索仓库是否已有可复用实现。**
- 能扩展现有函数则扩展；只有确实没有合适抽象时再新增。
- 报表、大屏、安装脚本中的聚合逻辑：优先复用或抽到单一数据访问层，避免三处复制粘贴后漂移。

## 实现策略：只保留一种稳定简洁写法

- 对同一需求，**只实现一种**最稳定、最简洁的方案，不并行保留多种写法「供选择」。
- 不写「以防万一」的第二条代码路径、不写未验证的备用实现。
- 避免过度抽象：一层函数能说清的，不要拆成深嵌套工具链。

## 禁止兜底式烂代码

- **不写笼统兜底**：如大段 `try/except: pass`、吞掉异常后假装成功、无意义的默认返回掩盖错误。
- **不写防御性垃圾**：对不可能出现的分支堆砌空判断、重复校验、魔术默认值。
- 需要容错时：明确失败条件，抛出或返回可读错误，让调用方能处理。
- 用户未要求时，不添加「兼容所有历史脏数据」的迁回逻辑。

## Frappe / 本项目约定

- DocType / Report / Page 的元数据与代码放在标准 App 路径下。  
- 开通 / 流失事实只进 `User Service Event`；报表与大屏只读聚合，不另写一套事实表。  
- 安装灌数走 `install.py` / `after_install`；不要在 Page JS 里写死业务样例库。  
- 白名单 API 保持薄：查库、聚合、返回结构清晰的 dict。  
- 不引入 ERPNext 依赖，除非用户明确要求。

## 文档与进度

- 完成阶段性工作后，更新 `PROGRESS.md`（做了什么、还剩什么、如何验证）。  
- 产品形态变更同步 `PRODUCT.md`；技术选型 / 模块边界变更同步 `ARTITECHTURE.md`。  
- 用户未要求时，不主动新增 markdown 文档。

## Git

- 仅在用户明确要求时 commit / push。  
- 不提交密钥、`.env`、本地 `frappe_docker` 运行时与 `frappe-bench` 数据。

## 协作语气

- 对用户说明简洁、直接；默认中文。  
- 不复述冗长过程；把结论、入口 URL、验证方式说清楚即可。
